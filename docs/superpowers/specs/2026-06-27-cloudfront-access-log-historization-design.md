# CloudFront Access-Log Historization — Design

**Date:** 2026-06-27
**Status:** Approved (design)
**Scope:** `packages/infrastructure` only. No application/Lambda/frontend changes.

> **Amendment (later):** delivery was switched from flat to **Hive-partitioned** via
> `s3_delivery_configuration { suffix_path = "{yyyy}/{MM}/{dd}"; enable_hive_compatible_path = true }`,
> so objects now land at `raw/app/year=YYYY/month=MM/day=DD/<auto>.parquet`. The hive flag
> MUST be `true` — only then does AWS allow the `key=value` layout and auto-expand the bare
> `{yyyy}/{MM}/{dd}` placeholders into `year=/month=/day=`; writing `year={yyyy}` literally
> with the flag off is rejected ("Provided suffixPath is invalid"). The delivery also needs
> `depends_on = [aws_s3_bucket_policy.cloudfront_logs]` (CreateDelivery validates write
> access, so the policy must exist first). Any "flat under `raw/app/`" / "no partitioning" /
> "no `s3_delivery_configuration` block" statements below are superseded. The base prefix is
> unchanged (`raw/app`), so the bucket, whole-bucket lifecycle, and whole-bucket policy
> `Resource` are unaffected.

## Goal

Capture the client IP and request metadata of every request to `meal-planner.isnan.eu`
and retain it for 90 days in S3. **Observe-only** — no blocking (WAF is explicitly out
of scope; can be added independently later). Just deliver the data to a bucket; no query
layer or dashboard is built or documented now (the Parquet data remains queryable later
if ever wanted, but that is out of scope).

Priorities, in order: (1) security / abuse investigation, (2) analytics / curiosity.
No user attribution — IP-level historization is sufficient.

## Why CloudFront access logs

CloudFront already fronts everything for this app: `/*` → S3 frontend, `/api/*` →
API Gateway. It sees the real viewer IP before any origin, so a single log stream on
the existing `aws_cloudfront_distribution.main` covers all paths with zero application
code changes.

## Decisions made during brainstorming

| Decision | Choice | Rationale |
|---|---|---|
| Act vs observe | **Observe only** | WAF (~$6/mo floor) deferred; additive later, no rework |
| Query layer | **None now — just deliver to S3** | Athena/dashboards are additive later; intentionally not built or documented now |
| Format | **Parquet** | Columnar; keeps future query/dashboard options cheap. Small CloudWatch conversion charge, negligible at this volume |
| S3 layout | **Hive date partitions under `raw/app/`** | Objects land at `raw/app/year=YYYY/month=MM/day=DD/`. The `app` segment namespaces this distribution so future log sources can use sibling prefixes; the `year=/month=/day=` suffix keeps partitions cheap to scan if ever queried. Specifying the `raw/app` prefix also suppresses CloudFront's default `AWSLogs/.../CloudFront/` path |
| Retention | **90 days, auto-delete** | S3 lifecycle rule; bucket self-prunes |
| Bucket topology | **One dedicated bucket for meal-planner** | S3 has no per-bucket fee; cost is identical to a shared bucket, but a per-app bucket gives clean blast-radius isolation and owner clarity |
| Bucket name | **`meal-planner-cloudfront-logs-<account-id>`** | Account-id suffix for global-uniqueness; chosen during brainstorming |
| File layout | **One new self-contained `logs.tf`** | meal-planner has no `s3.tf` (its buckets live in `webclient.tf`); keeping bucket + delivery wiring together gives a clean blast radius for a self-contained feature |
| Logging version | **Standard logging v2** | Keeps S3 ACLs disabled (AWS-recommended, bucket-policy based); supports Parquet output and a custom flat prefix |

## Architecture

```
                          viewer request (real client IP)
                                      │
                                      ▼
                       aws_cloudfront_distribution.main
                                      │
              standard logging v2 (CloudWatch Logs Delivery API,
                      registered in us-east-1)
                                      │
                                      ▼
        ┌──────────────────────────────────────────────────┐
        │  delivery_source (ACCESS_LOGS) ──► delivery ──►    │
        │  delivery_destination (S3, parquet)               │
        └──────────────────────────────────────────────────┘
                                      │
                                      ▼
             S3: meal-planner-cloudfront-logs-<account-id> (eu-central-1)
             └── raw/app/year=YYYY/month=MM/day=DD/<auto-generated>.parquet
                                      │
                       (90-day lifecycle expiration)
```

The leaf filename (`<auto-generated>.parquet`) is produced by AWS's vended-log delivery
service (timestamp + unique id); it is not controllable, and many files are written per
period. We control the `raw/app/` base prefix and the `year=/month=/day=` partition
layout via `suffix_path`.

## Components

### 1. S3 log bucket (new `logs.tf`, default eu-central-1 provider)

- `aws_s3_bucket` `meal-planner-cloudfront-logs-<account-id>`, `force_destroy = true`.
- `aws_s3_bucket_public_access_block` — all four flags `true`.
- `aws_s3_bucket_server_side_encryption_configuration` — `AES256` (SSE-S3). (v2 delivery
  to S3 supports SSE-S3; SSE-KMS would require extra key-policy grants — out of scope.)
- `aws_s3_bucket_lifecycle_configuration` — one rule, `Enabled`, **whole bucket**
  (`filter {}`, no prefix), `expiration { days = 90 }`, plus
  `abort_incomplete_multipart_upload { days_after_initiation = 3 }`. The bucket is
  dedicated to these logs (everything lives under `raw/`), so expiring everything is
  correct and simplest.
- `aws_s3_bucket_policy` — grant the log-delivery service principal write access
  (matches AWS's documented `AWSLogsDeliveryWrite` statement):
  ```
  Principal: { Service: "delivery.logs.amazonaws.com" }
  Action:    "s3:PutObject"
  Resource:  "<bucket-arn>/*"            # whole bucket — avoids path-prefix mismatch
  Condition:
    StringEquals:
      "s3:x-amz-acl":      "bucket-owner-full-control"
      "aws:SourceAccount": <this account id>
    ArnLike:
      "aws:SourceArn": "arn:aws:logs:us-east-1:<account>:delivery-source:*"
  ```
  **Gotcha (known failure mode):** if the `aws:SourceAccount` / `aws:SourceArn` /
  `s3:x-amz-acl` conditions are missing or wrong, or the `Resource` path doesn't cover
  where logs land, delivery **silently fails** with `AccessDenied` — logs never appear
  and nothing is surfaced on the distribution. Reuse the existing
  `data.aws_caller_identity.current` / `local.account_id` (declared in `iam.tf`) for the
  account id; whole-bucket `Resource` sidesteps the prefix-mismatch trap AWS documents.

### 2. Log delivery (v2) — all three resources use `provider = aws.us_east_1`

The CloudWatch Logs Delivery API for CloudFront must be called in `us-east-1` even
though the destination bucket is in `eu-central-1` (cross-region delivery is allowed).
The `aws.us_east_1` alias already exists in `main.tf` (used for the ACM cert).

> **Naming note:** these resources are prefixed `aws_cloudwatch_log_delivery_*` even
> though the source is CloudFront and the destination is S3 (CloudWatch stores nothing
> here). "Log delivery" is a generic CloudWatch Logs subsystem that routes logs from
> many AWS sources to S3/Firehose/CloudWatch/X-Ray; CloudFront's docs call this same
> feature "standard logging v2." The `cloudwatch` prefix is just the API family that
> operates the pipe — logs land as Parquet in S3, not in CloudWatch.

- `aws_cloudwatch_log_delivery_source` `meal_planner_cloudfront`
  - `log_type    = "ACCESS_LOGS"`
  - `resource_arn = aws_cloudfront_distribution.main.arn`
- `aws_cloudwatch_log_delivery_destination` `meal_planner_cloudfront_s3`
  - `delivery_destination_type = "S3"`
  - `output_format             = "parquet"`
  - `delivery_destination_configuration { destination_resource_arn = "<bucket-arn>/raw/app" }`
    — the `/raw/app` prefix in the ARN suppresses CloudFront's default
    `AWSLogs/aws-account-id=<id>/CloudFront/` path; logs land under `raw/app/`.
- `aws_cloudwatch_log_delivery` `meal_planner_cloudfront`
  - `delivery_source_name     = <source>.name`
  - `delivery_destination_arn = <destination>.arn`
  - `s3_delivery_configuration { suffix_path = "{yyyy}/{MM}/{dd}"; enable_hive_compatible_path = true }`
    — appends Hive-style date partitions under `raw/app/`. The hive flag MUST be `true`:
    only then does AWS allow the `key=value` layout and auto-expand the bare
    `{yyyy}/{MM}/{dd}` placeholders into `year=/month=/day=`. Writing `year={yyyy}` literally
    with the flag off is rejected ("Provided suffixPath is invalid").
  - `depends_on = [aws_s3_bucket_policy.cloudfront_logs]` — CreateDelivery validates write
    access at creation, so the bucket policy must already exist or delivery creation fails.
  - `record_fields` — the 14-field set (see §3)

### 3. Record fields (the data we actually keep) — **14-field set (chosen)**

```
date, time, c-ip, c-country, asn,
cs-method, cs-protocol, cs(Host), cs-uri-stem, cs-uri-query,
sc-status, x-edge-result-type, x-edge-location, cs(User-Agent)
```

`c-ip` is the whole point (client IP). The rest give enough context to tell a scanner
from a real visitor and to group by path/status. The full CloudFront v2 field list is
available if more is wanted later — adding fields is a one-line change.

**Exact field names matter** — they are validated at apply time. `date`/`time` (there is
no single `timestamp` standard field); Host and User-Agent use the W3C parenthesized
form `cs(Host)` / `cs(User-Agent)`; the rest are hyphenated lowercase.

**Geo note:** standard logging **v2** can select a subset of real-time fields, so
geography is available natively per row — no header injection, no IP lookup:
- **`c-country`** — ISO 3166-1 alpha-2 country code, derived by CloudFront from the
  viewer IP (e.g. `FR`, `US`).
- **`asn`** — viewer's Autonomous System Number (ISP / network owner); distinguishes
  datacenter/VPN/hosting traffic from residential — a strong bot/scanner signal.
- `x-edge-location` — serving POP (e.g. `FRA56`); rough-region hint.

Precise *city* still needs an IP→geo lookup later, but country + network are columns in
the logs.

## Data flow

1. Viewer hits `meal-planner.isnan.eu`; CloudFront records the request (real `c-ip`).
2. Standard logging v2 batches and delivers Parquet files to
   `s3://meal-planner-cloudfront-logs-<account-id>/raw/app/year=YYYY/month=MM/day=DD/`
   (delivery latency: minutes, not real-time).
3. Lifecycle rule deletes objects 90 days after creation.

## Error handling / operational notes

- **Silent delivery failure** is the main risk — verify after `apply` that Parquet
  objects actually appear under `raw/app/` within ~15 min of generating traffic.
  If empty, the bucket policy conditions/`Resource` are the first suspect.
- **Cost:** pennies/year at this app's volume. S3 storage of a few MB/month of Parquet,
  plus a small CloudWatch *conversion* charge for the Parquet output (negligible here).
- **Teardown:** `force_destroy = true` lets `terraform destroy` remove the bucket with
  objects present. Per-app bucket means destroying meal-planner cleanly removes its logs
  and nothing else.

## Out of scope

- AWS WAF / blocking / rate limiting (additive later; ~$6/mo floor).
- User-to-IP attribution.
- Any query layer or dashboard (Athena, QuickSight, Grafana, custom page) — the Parquet
  data remains queryable later if wanted, but nothing is built or documented now. (The
  `year=/month=/day=` partitions make a future Athena table cheaper, but none is built.)
- Shared multi-app log bucket (rejected in favor of per-app isolation).

## Verification

1. `terraform -chdir=packages/infrastructure plan` — review the new bucket + 3 delivery
   resources (all in us-east-1) + bucket policy.
2. `apply`, generate a few requests against `meal-planner.isnan.eu`.
3. Within ~15 min, confirm Parquet objects under
   `s3://meal-planner-cloudfront-logs-<account-id>/raw/app/` (in `year=/month=/day=`
   partition folders).
4. Spot-check one file contains a populated `c-ip` / `c-country` column.
